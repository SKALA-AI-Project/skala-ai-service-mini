# AGENTS.md

이 저장소의 기본 작업 언어는 한글이다.
구현 전에 요청 유형을 판단하고 아래 문서로 먼저 이동한다.

## 공통 규칙

- 최종 목표는 HBM4, PIM, CXL 기술 전략 PDF 보고서 자동 생성이다.
- 아키텍처는 Supervisor 중심 순차 워크플로우를 따른다.
- 최근 3개월 이내 근거를 우선 사용한다.
- 모든 주장에는 추적 가능한 출처가 필요하다.
- TRL 4~6은 반드시 "추정"과 한계를 함께 표기한다.
- 재시도는 단계별 최대 2회까지만 허용한다.
- 매 작업마다 `design.md` 요구사항과 현재 구현 결과를 1:1로 대조 검증한다.
- 검증에서 누락이나 불일치가 나오면 수정 후 다시 검증하는 피드백 루프를 수행한다.
- 구현 코드는 개발자가 바로 이해할 수 있는 수준의 간결한 주석을 포함한다.

## 라우팅

- 프로젝트 목적과 범위를 파악할 때: `contexts/01_프로젝트개요.md`
- 워크플로우와 성공 기준을 볼 때: `contexts/02_워크플로우.md`
- 에이전트 역할과 입출력을 볼 때: `contexts/03_에이전트정의.md`
- TRL 판정 방식과 신뢰도 규칙을 볼 때: `contexts/04_trl방법론.md`
- 상태값과 데이터 계약을 볼 때: `contexts/05_상태정의.md`
- 보고서 구조와 산출물 기준을 볼 때: `contexts/06_보고서규격.md`
- 구현 단계와 일정 관리를 볼 때: `PLAN.md`
- 폴더 구조와 파일 배치를 볼 때: `DIRECTORY_STRUCTURE.md`

## 작업 지침

- 새 기능 구현 전 `DIRECTORY_STRUCTURE.md`에 맞는 위치를 먼저 확인한다.
- 프롬프트 변경 전 보고서 섹션 규격과 품질 기준을 함께 확인한다.
- 검색 로직 수정 시 확증 편향 방지 규칙과 HITL 조건을 유지한다.
- Supervisor, Search, TRL, Draft, Formatting 책임을 혼합하지 않는다.
- 정보가 부족하면 추정하지 말고 부족함을 명시한다.
- 작업 완료 전 `design.md` 항목별로 구현 산출물을 대응시켜 빠진 항목이 없는지 점검한다.
- 검증 결과는 다음 작업의 입력으로 반영하고, 미충족 항목이 있으면 먼저 보완한다.
